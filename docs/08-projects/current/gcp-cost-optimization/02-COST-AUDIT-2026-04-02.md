## Root Cause Analysis

### Root Cause #1: Pipeline Canary — 3,600+ BQ Queries Every 15 Minutes

The `nba-pipeline-canary` Cloud Run Job runs on a `*/15 * * * *` schedule (every 15 minutes). Each run generates **3,600–3,800 BigQuery queries**, including:

| Query Pattern | Count/Day | GB/Day |
|---------------|-----------|--------|
| `espn_team_rosters` per-player lookup | 82,890 | 6.1 |
| Team defensive rating per-player | 55,260 | 0.03 |
| FT attempts/pace per-player | 55,260 | 0.03 |
| Repro check (`nba_orchestration.reprocessing`) | 28,690 | 0.02 |
| Team pace per-player | 27,630 | 0.02 |
| Travel/jet lag per-player | 7,549 | 0.00 |

**Impact:**
- BQ from compute SA: 7.8M queries/month → **$65/mo**
- Cloud Run Jobs CPU: **$15/mo**
- **Total canary cost: ~$80/mo**

The canary is running per-player feature queries at monitoring frequency — essentially recomputing the feature store every 15 minutes as a side effect of health checks.

### Root Cause #2: Phase 3 Analytics at 4 CPUs, Pub/Sub-Triggered

`nba-phase3-analytics-processors` has 4 CPUs and is triggered by every Pub/Sub message from Phase 2 completions. With 40+ scrapers firing throughout the day, it runs frequently at high CPU allocation.

**Cost: $125/mo** — the single biggest NBA Cloud Run service.

### Root Cause #3: `infinite-case` Backend on Same Billing Account

`infinitecase-backend` service in the `infinite-case` project:
- 16 GiB RAM, 4 CPU, `min-instances=1` → billing 24/7
- Actively deployed (84 revisions, last deployed 2026-04-01 at 3:41 AM)
- **$115/mo** for a non-NBA project sharing the billing account

### Root Cause #4: Cloud Logging Volume — 186 GB/Month

91 Cloud Run services all logging to Cloud Logging with no exclusion filters. Free tier: 50 GB/project. Overage: 136 GB × $0.50 = **$68/mo**.

### Root Cause #5: Artifact Registry — 111 GB of Stale Docker Images

| Repo | Size |
|------|------|
| nba-props | 111 GB |
| cloud-run-source-deploy | 17 GB |
| gcf-artifacts | 16 GB |

| mlb-validators | 0.8 GB |

No lifecycle policy. Each deploy adds new image layers without pruning old ones. **$23.75/mo**.

### Root Cause #6: 167 Cloud Scheduler Jobs

GCP charges $0.10/job/month (3 free per billing account). 167 jobs = 164 paid = **$16.40/mo**. Prior audit (Session 225) found 110 jobs and targeted ~30-40. Service count has grown from 73 → 91 Cloud Run + new MLB schedulers.

## Action Plan

### P0 — No Risk, Easy Wins (~$120/mo savings)

- [ ] **Reduce canary frequency: 15min → 60min**
  - Edit: `nba-pipeline-canary-trigger` scheduler job
  - Change cron from `*/15 * * * *` to `*/60 * * * *`
  - Saves: ~$60/mo (75% reduction in BQ + Cloud Run Jobs cost)
  - Risk: 1-hour max lag on canary alerts (acceptable — we have Slack alerts + self-heal)

- [ ] **Add Cloud Logging exclusion filters**
  - Exclude INFO + DEBUG logs from these high-volume services: `nba-phase2-raw-processors`, `nba-phase3-analytics-processors`, `nba-phase4-precompute-processors`, `nba-scrapers`
  - Keep ERROR + WARNING everywhere. Keep all logs for `prediction-coordinator`, `prediction-worker`.
  - Saves: ~$40/mo (target: drop from 186 GB to ~50 GB = in free tier)
  - GCP Console → Logging → Log Router → Exclusions

- [ ] **Delete `nba-phase1-scrapers`** (confirmed legacy, superseded by `nba-scrapers`)
  - `gcloud run services delete nba-phase1-scrapers --region=us-west2 --project=nba-props-platform`
  - Saves: ~$0.24/mo (negligible) but reduces scheduler noise
  - Risk: None — CLAUDE.md confirms this is dead

### P1 — Medium Impact, Low Risk (~$100/mo savings)

- [ ] **Reduce Phase 3 analytics CPUs: 4 → 2**
  - `nba-phase3-analytics-processors`: change CPU limit from 4 to 2
  - Analytics processing runs longer but at half the cost
  - Saves: ~$60/mo
  - Risk: Low — run at current load first. Phase 3 is triggered by Pub/Sub, not time-critical.
  - Validate: p95 latency before changing, confirm Phase 4 timeout is sufficient

- [ ] **Prune Artifact Registry old image tags**
  - Keep last 3 tags per service in `nba-props` repo (currently 111 GB)
  - Use `gcloud artifacts docker images list` + `delete` or add a lifecycle policy
  - Target: reduce from 111 GB to ~15 GB = saves ~$10/mo
  - Also prune `cloud-run-source-deploy` (17 GB) and `gcf-artifacts` (16 GB)
  - Total saves: ~$13/mo

- [ ] **Audit Cloud Functions min-instances**
  - Identify which CFs have min-instances set (cost: $44/mo in warmup overhead)
  - Remove min-instances from CFs that don't need sub-second cold-start: monitoring CFs, cleanup CFs, weekly batch CFs
  - Keep min-instances on latency-sensitive CFs (phase transitions, orchestrators)
  - Saves: ~$20/mo

### P2 — Lower Priority (~$25/mo savings)

- [ ] **Audit and reduce Cloud Scheduler jobs (167 → ~80)**
  - 167 jobs, 164 paid = $16/mo
  - Review jobs tied to deleted or renamed services
  - Saves: ~$8/mo

- [ ] **Delete stale dev/staging services**
  - `prediction-coordinator-dev`: 1Gi/1CPU, not min-instances but idle cost
  - Any `*-staging`, `*-dev`, `*-test` services confirmed unused
  - `gcloud run revisions list` to check last invocation time
  - Saves: ~$5/mo

### Separate Decision — `infinite-case` Backend ($115/mo)

The `infinitecase-backend` service is an actively developed project (84 revisions, deployed today). Setting `min-instances=0` would save **$115/mo** but adds cold-start latency on first request.

Options:
1. **Set min-instances=0** — saves $115/mo, first-request cold start ~5-10s
2. **Downsize to 4Gi/2CPU** — saves ~$57/mo, still always-warm
3. **Keep as-is** — $115/mo, zero latency
